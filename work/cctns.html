CCTNS — Ministry of Home Affairs draft

A national policing platform connecting citizens, officers, and state machinery — designed for India's 1.4 billion. End-to-end FIR management, citizen services, and security operations on one stack.

TL;DR

  • Designed a single citizen-facing app spanning FIR filing, complaint tracking, and emergency services across 28 states + 8 UTs.
  • Built an officer dashboard architecture that consolidates fragmented state-level CCTNS instances into one consistent OS.
  • Led a team of 5 designers, established the design system + governance model, shipped the first pilot rollout in [N] states.

The constraint

Designing for India's full citizen base means designing for extreme variance at every axis — language, literacy, device, bandwidth, trust. A user could be a metro lawyer filing online from an iPhone, or a rural farmer dictating a complaint in Marathi over a 2G connection at a village outpost.

Layered on top: 28 state police forces, each with their own legacy systems, terminology, and political ownership. Any UI we built would be vetoed if it didn't respect state-specific workflows. And being a Ministry of Home Affairs project, every screen had to clear security, accessibility (RPwD Act), and data-localisation review.

Six languages at launch (Hindi, English + 4 regional). Offline-first by default. Officers without smartphones using browser kiosks. Citizens whose first contact with government services is this app.

The decision

The core call was whether to build one citizen experience that abstracts state differences, or a federated experience where each state customises. We picked the first — at significant political cost — because anything else would replicate the existing fragmentation that made CCTNS a problem in the first place.

Chose
A unified citizen surface with state-specific logic in the backend. One UI vocabulary across India.
Killed
Per-state UI templating. Dropped because it would've calcified state silos forever.
Chose
Voice-first citizen flows for low-literacy users, not as an "accessibility feature" but as the primary path.
Chose
A mentorship model where each junior designer owned one citizen vertical (FIR, missing person, women's safety, traffic, cyber).
Killed
Real-time officer-citizen chat. Compliance review couldn't approve unfiltered messaging on a national platform.

The system

The platform's backbone is a three-tier design system — semantic tokens at the core, vertical-specific component variants in the middle, state-customisation slots at the surface. This lets us ship one citizen app while letting states adapt language, terminology, and minor flow steps without forking the codebase.

Component scope at launch: ~80 components, 5 verticals, 6 languages, all wired through Tokens Studio with an automated pipeline into Storybook for engineers.

AI in the loop

This project is where I started leaning hard into AI-assisted design workflows. Three concrete uses:

  • Conversation flow generation: drafting voice-first flows in Hindi + 4 regional languages would've taken weeks of per-language work. I built (and later open-sourced) a Figma plugin that uses Anthropic's models to generate localised flow variants from a single English source.
  • System validation: with 80+ components × 6 languages × 5 verticals, manual review of token usage was untenable. The Design System Validator plugin I built flags structural breaks before engineering handoff — caught ~40% of issues that would've otherwise reached PR review.
  • Claude Code for prototyping: citizen flows that needed real-time interaction (kiosk mode, voice playback) were prototyped in code, not Figma. The CLAUDE.md Exporter plugin lets Claude Code build those prototypes inside our system tokens automatically.

Where I didn't use AI: any flow involving evidence, complainant identity, or officer judgement. Designing those with AI in the loop would've been irresponsible at this scale.

Trade-offs

The unified UI decision cost us political capital with two state IT departments who wanted their own branding on the citizen app. We absorbed that — escalated, lost a battle, kept the larger war. The system survives.

Voice-first flows are slower for power users (lawyers, NGOs filing repeatedly) than a dense form. We accepted that. The platform's ground-level legitimacy depends on the rural farmer being able to use it, not the lawyer being optimised for.

What I'd change

I would have invested earlier in field research with non-metro officers. We did the citizen-side ethnography well, but our officer assumptions were Delhi-coded for the first quarter, and we burned cycles redesigning officer flows once we got to ground truth in [state]. The lesson stuck: in regulated multi-stakeholder work, every type of user needs first-hand contact, not just the headline one.

Also: I'd have written the governance model on day one, not month three. We had drift across the team's outputs until we sat down and articulated "what counts as a system decision vs. a vertical decision." Cost us two weeks of consolidation.
