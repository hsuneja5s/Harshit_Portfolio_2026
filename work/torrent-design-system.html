Torrent Diagnostic — Brand & Design System

Cutting design inconsistency with a reusable brand and UI system. One identity, one component library — powering the patient app, the phlebotomist field app, and the clinical web. Built from scratch.

TL;DR

  • Built a brand identity + design system foundation from scratch — used across 3 distinct product surfaces under the Torrent Diagnostic umbrella.
  • Established semantic token architecture (background / content / border × primary / secondary / tertiary) so any surface can theme without touching component internals.
  • Cut design inconsistency at the source — engineers across product teams now compose from the same library instead of inventing their own components.

The constraint

Torrent Diagnostic was launching three products in parallel — the consumer-facing Patient App, a field-collection app for phlebotomists, and a clinical-side web tool. Three different product teams. Three different audiences. Three different platforms.

Without a shared system, we'd ship three products that visibly belonged to three different companies — even though they all carried the Torrent name. Worse: every product team would solve the same component problems independently, badly, and slowly.

Hard constraint: no luxury of a "freeze and build the system" phase. The patient app and phlebotomist app were on overlapping launch timelines. The system had to ship alongside the products, not before them.

The decision

The architectural call: brand-first, then system, then components. Most design systems do this in the wrong order — components first, then add system rules, then retrofit brand. We inverted it because the brand was the part that had to flex across audiences.

Chose
A semantic token architecture from day one — even though it costs more upfront than literal tokens. Healthcare audiences vary widely (senior consumers, working phlebotomists, clinical staff) and each needs a different visual emphasis without forking the system.
Chose
Brand identity that prioritises calm + clarity over "diagnostic clinical." Healthcare branding often defaults to sterile blues and whites. Torrent's audience needed warmth — a daily companion, not a hospital interface.
Chose
A core component library scoped to ~40 components — the 80% used everywhere. Anything beyond was a "vertical extension" owned by the consumer team, not the system team.
Chose
Strict accessibility baseline at the system level — 44pt tap targets, AA contrast minimum, voice-readable text labels — so every surface inherited it without thinking about it.
Killed
Per-surface component variants. Tempting (the patient app uses different button sizing than the phlebotomist app), but governance debt forever. Solved instead with token-level theming on shared components.
Killed
A separate "brand-only" Figma file disconnected from the component library. Brand and system stay in the same source of truth — drift between them is the most common system failure I've seen.

The system

Three layers. Brand at the bottom (logo, palette, typography, tone). System in the middle (semantic tokens, primitive components). Surface at the top (the patient app's home screen, the phlebotomist app's task flow, the clinical web's dashboards).

Tokens flow through Tokens Studio → Style Dictionary → engineering. Designers change a brand token in Figma; engineering picks up the diff via PR. No hand-syncing, no documentation drift between Figma and code.

AI in the loop

The system gave me the testing ground for the AI Figma plugins I now use across PwC engagements. Specifically:

  • Design System Validator: with three product teams shipping in parallel, drift was inevitable without automated checks. The Validator catches token mismatches and structural breaks before they reach engineering.
  • Design System Generator: the Torrent identity-to-system pipeline became the template for this plugin — input one brand color + a few choices, get a starter system. Now reusable for any new internal Torrent product.
  • CLAUDE.md Exporter: when engineering started using Claude Code to scaffold new admin tools, they needed the system context Claude could read. Built the exporter, gave it to them, drift dropped.

AI doesn't make the design system; the design system constrains AI's output to be system-aligned by default. That's the loop.

Trade-offs

Concurrent-with-products meant the system had inconsistencies in the first quarter — tokens shifted as we learned what worked. Engineers got frustrated. We mitigated with a "what's stable" indicator on every component, but it was real friction.

Choosing semantic tokens over literal slowed first-component velocity. The patient app's launch could have been a few weeks earlier if we'd shipped literal tokens. We absorbed the delay because the system is an asset that lives for years; the launch is a moment.

A 40-component scope means some patterns (specialised data tables, complex forms) had to be built outside the system in v1. Those are now system v2 candidates — the long tail of every design system.

The Patient App and Phlebotomist App both shipped on top of the system. The clinical web is in flight. The next product Torrent launches will start from this foundation in a fraction of the time the first three took.

What I'd change

I'd have shipped contribution tooling from day one — PR templates, automated changelogs, RFC process for component proposals. We added them in month four. Done at month one, the contribution velocity from the consumer teams would have been double.

And: I'd have invested in a system Storybook earlier. We had Figma for the design layer but engineering's component documentation lagged behind for the first quarter. A live Storybook from day one would have been a forcing function for parity.
