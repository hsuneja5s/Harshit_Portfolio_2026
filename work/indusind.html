IndusInd Bank — Design System draft

A design system from the ground up — replacing a fragmented product experience across mobile and web with one unified banking application.

TL;DR

  • Built a semantic-token-first design system from scratch — one source of truth for mobile, web, and internal tools.
  • Established governance + contribution model so the system survives without me. Engineers and designers ship new components without breaking consistency.
  • Replaced years of UI drift with a unified banking experience. Component reuse went from nominal to ~85% across the app.

The constraint

IndusInd had years of organic product growth across multiple feature teams. Mobile and web had drifted apart. The same "transfer money" flow looked, behaved, and was named three different ways across surfaces. Customers noticed. So did regulators, who flagged inconsistency in disclosures.

The mandate: build a system that brings the surfaces together — without forking the migration into a multi-year nightmare. We had to ship the system and migrate consumers to it concurrently. No "freeze the product, build the system" luxury.

The decision

The architectural call was semantic tokens, not literal tokens — even though semantic tokens cost more upfront. Banking has long surface-level requirements (dark mode, regional themes, white-label for partner banks) that literal tokens couldn't have absorbed.

Chose
Semantic token architecture: background / content / border × primary / secondary / tertiary. Engineers theme by remapping a single layer.
Killed
Literal tokens (color-blue-500 etc.). Faster to ship, impossible to theme. Saying no early was the highest-leverage decision.
Chose
Component-by-component migration with explicit "until consumed" deprecation tags. Old components kept working, but every new feature had to use the system.
Chose
A contribution model where consumer designers can propose new components, but a small architecture committee reviews. Prevents drift without bottlenecking.
Killed
A "lite" version of the system for non-banking surfaces. Adds maintenance burden, gives nothing back.

The system

Three layers, single source of truth: tokens in Tokens Studio (Figma) sync to Style Dictionary (CI/CD pipeline) which generates platform-specific outputs (CSS variables for web, XML for Android, JSON for React Native).

The governance doc is short on purpose: "any change that affects more than one team's UI is a system change." That sentence has prevented more drift than any tooling we built.

AI in the loop

This is the project where I built the AI Figma plugins that now sit in my plugin shelf. Each one was a real pain-killer first, a portable tool second.

  • Design System Validator: caught 100s of token-mismatch + structural breaks before they landed in PR. Built because manual review across 6 consumer designers wasn't scaling.
  • CLAUDE.md Exporter: when engineers wanted to scaffold prototypes via Claude Code, they kept generating components that didn't match our tokens. The exporter packages our system as a single CLAUDE.md so Claude builds inside the system by default.
  • Design System Generator: we forked it for the IndusInd internal-tools team — instead of waiting weeks for system support, they could spin up a token-aligned starter system for a new internal app in a day.

Every plugin started as a "I'm doing this manually 30 times a week" moment. None were built as side projects in isolation; all were built for this team and the work itself.

Trade-offs

Choosing semantic tokens cost us a quarter of upfront design + engineering work compared to literal. We absorbed it. Two years from now when dark mode lands in a week instead of a quarter, that bill will be paid back many times over.

Concurrent migration meant some features shipped during the transition were temporarily inconsistent — a "what's on the system" sticker on every Figma frame helped engineers know what they could trust. Imperfect, but better than freezing product velocity.

What I'd change

I'd have written the governance doc on day one, not month three. We had drift across the team's outputs until we sat down and articulated "what counts as a system decision vs. a vertical decision." Cost us ~2 weeks of consolidation. The doc itself is six bullet points; writing it earlier would have been free.

And: I'd have invested in contribution tooling (PR templates, automated change-logs) sooner. We did them in month four. Done at month one, the contribution velocity would've been double.
